## I1 — Orchestrator turn-state model (media-agnostic)

**Finding: there is NO "active specialist" lock across turns. Routing re-runs from scratch every turn.**

- Each user message → `CommandCenter.send()` → `POST /api/orchestrate` with `condenseForOrchestrator(newMessages)` ([`CommandCenter.tsx:649-658`](../../apps/web/app/components/CommandCenter.tsx)).
- `condenseForOrchestrator` ([`CommandCenter.tsx:102`](../../apps/web/app/components/CommandCenter.tsx)): keeps user turns verbatim, **drops all non-`isOutput` assistant stubs** ("so they can't anchor the router"), and **truncates each specialist deliverable to a 240-char excerpt**. Design intent (comment at `:643-648`): make "the LATEST user request drive the decision" and *avoid* anchoring the router to the previous department.
- `/api/orchestrate` → `runOrchestrator({ intent:"route", context:{ messages, message: lastUserMessage } })` ([`orchestrate/route.ts:62`](../../apps/web/app/api/orchestrate/route.ts)). It re-derives `department` + `task` every call and emits a `READY:{department,agentId,task,...}` marker.
- There is **no mechanism that recognizes "this user message is a reply to specialist X's question" and routes back to X with the question context.** The orchestrator sees a 240-char excerpt of the prior turn but is explicitly tuned to weight the latest message.

Applies to ALL media: a video-brief follow-up has the exact same continuity gap as a logo-brief follow-up.

## I2 — Specialist context propagation (media-agnostic) — **PRIMARY ARCHITECTURAL DEFECT**

**Finding: the specialist invocation is STATELESS per turn. The Anthropic call receives a single-turn message with only the current `task` string — no conversation history.**

- `runAgent` ([`CommandCenter.tsx:814`](../../apps/web/app/components/CommandCenter.tsx)) POSTs to `/api/agent` with **only** `{ task, department, agentId, userId, pbToken, clientId, threadId }` — **the `messages`/thread is NOT sent** ([`:845-857`](../../apps/web/app/components/CommandCenter.tsx)).
- `/api/agent` builds the LLM call as `messages: [{ role:"user", content: task }]` ([`agent/route.ts:385`](../../apps/web/app/api/agent/route.ts)) — **one turn, no history.**
- Conversation turns ARE written to the `conversations` collection (`writeConversationTurnAndEnqueue`, [`agent/route.ts:31,293,411`](../../apps/web/app/api/agent/route.ts)) but are **never read back as thread history** into the prompt. Continuity is attempted only through system-prompt blocks: vault (`fetchVault`), voice (`getVoiceBlock`), semantic **LIVING MEMORY** (`retrieve(userId, task,…)` — keyed on the *current task*, [`:205`](../../apps/web/app/api/agent/route.ts)), and a legacy "last-2 same-dept documents" block ([`:258-283`](../../apps/web/app/api/agent/route.ts)).
- **None of these reliably contain the specialist's just-asked question.** The turn-N question is written + enqueued **async** (Vault ingestion is queued), so it is not yet indexed when turn N+1 runs; and semantic retrieval keys off the *answer* text, not the question.

**Consequence:** when a specialist asks a clarifying question (turn N) and the user answers (turn N+1), the specialist has **no memory of its own question**. The short answer ("logo", "for a real-estate listing") arrives with zero question-context. This is media-agnostic and is the core of the "turn-state amnesia."

## I3 — Natural-language generation trigger paths (ALL media)

**Finding: generation is CHIP-GATED for every media type. There is NO ungated natural-language → generation path. The "F5/F6 bypass" H2 sought does not appear to exist.**

- A free-text "generate visual / make a video / create the logo" message does **not** generate anything directly. It goes through `/api/orchestrate` → routes to a specialist → the specialist produces a **text deliverable** (for visuals, a dense prompt/brief).
- Image/video generation is triggered **separately** by the user clicking a `generate_image` / `generate_video` **action-affordance chip** → `openGenTier()` ([`CommandCenter.tsx:421`](../../apps/web/app/components/CommandCenter.tsx)) → `GenerationTierInline` modal (video) or 3-options (image) → `runGeneration` → `/api/integrations/muapi`. Standard #38 (`trigger-surfaces.ts`) enforces that every `runGeneration`/`runEdit` call site is gated.
- The **only** free-text → generation path in the codebase is the new edit-as-intent gate (`send()` → `/api/generation/edit`), and it (a) fires only when an artifact is the active edit target, and (b) is #38-registered. Not a bypass.
- **No code parses specialist output into a generation call.** The only markers parsed from streamed output are `READY:{…}` and `EXECUTE:{…}` ([`CommandCenter.tsx:684,1042`](../../apps/web/app/components/CommandCenter.tsx)) — both **routing/execution control, not media generation**.
- Image and video share the same trigger path (chip → `openGenTier` → tier-gate → `runGeneration`); video adds the tier picker, image skips to 3-options. No per-media divergence in the trigger mechanism.

**Honest note (possible wrong premise):** the brief frames an "ungated path H2 was supposed to find." Evidence says generation is uniformly chip-gated; I found **no** ungated path for image or video. ## I4 — "Something went wrong" source (media-agnostic) — **THE MASKING**

The screenshot's error is produced at THREE layers, each reducing a real error to a generic string:

1. **Server:** [`agent/route.ts:433-436`](../../apps/web/app/api/agent/route.ts) — the outer `catch` does `console.error("Agent route error:", err)` then `return new Response("Something went wrong", { status: 500 })`. **The real error IS captured in the Vercel function logs; the client never sees it.**
2. **Client (agent phase):** [`CommandCenter.tsx:859`](../../apps/web/app/components/CommandCenter.tsx) `if (!res.ok) throw "Agent failed"` → caught at [`:887-897`](../../apps/web/app/components/CommandCenter.tsx) → message replaced with **"Something went wrong. Try again."** ([`:895`](../../apps/web/app/components/CommandCenter.tsx)).
3. **Client (orchestrate phase):** [`CommandCenter.tsx:708-714`](../../apps/web/app/components/CommandCenter.tsx) — any throw in the orchestrate try (incl. `!res.ok` → "Orchestrate failed") → **"Something went wrong. Try again."** ([`:711`](../../apps/web/app/components/CommandCenter.tsx)).

`runOrchestrator` is wrapped to **never throw** (returns a degraded envelope, [`orchestrator/index.ts:47-60`](../../apps/web/app/api/_lib/orchestrator/index.ts)), so the orchestrate route rarely 500s. **The most likely crash site is `/api/agent` throwing (site #1 → #2).**

**H1 relationship:** H1's fail-loud is in the **muapi** route, which is NOT on the agent/orchestrate path — so H1 is **not** directly the agent crash. The masking that hides this diagnostic is the generic `"Something went wrong"` at sites #1–#3, not H1.

## ROOT CAUSE STATEMENT

**The failure is conversational-pipeline-wide (affects ALL media types), not image-specific.**

Two compounding architectural defects + one masking layer:

1. **Turn-state amnesia (I1+I2, media-agnostic):** the pipeline has no per-thread conversational memory at the specialist layer. The orchestrator re-routes every turn from a 240-char-truncated view, and the specialist runs `messages:[{role:"user",content:task}]` with **no history**. A user's answer to a specialist's question reaches the specialist stripped of the question. This degrades or breaks EVERY multi-turn specialist interaction — logo, video, anything.
2. **A runtime throw in `/api/agent` (I4, media-agnostic):** something in the agent handler throws on the failing turns, producing a 500. The exact throw is **logged but masked** and not yet identified (needs the Vercel log).
3. **Diagnostic masking (I4):** three layers reduce the real error to "Something went wrong," so neither operator nor user can see the cause — for any media type.

"No visual generated" is **downstream** of (2): if the specialist crashes, it never emits the prompt, so the "Generate Image" chip never appears and nothing can be generated. There is **no ungated/auto generation path** (I3) — so the fix is NOT "add a NL→generate path"; it's "make the specialist turn succeed + carry context."

**Standard #32 conflation guard:** a fix that only touches image generation would NOT address this root cause. The root cause is the conversational pipeline (orchestrator turn-state + agent statelessness + error masking), shared by image and video alike.

## PROPOSED FIX SCOPE (architecture only — NOT code; covers image AND video)

1. **Thread-history propagation (closes I2, all media):** pass real conversation history into the specialist call — either send the condensed thread from `runAgent`, or have `/api/agent` load the last N turns of `threadId` from the `conversations` collection and build a multi-turn `messages` array (not single-turn). Media-agnostic by construction.
2. **Reply-aware routing (closes I1, all media):** when the prior assistant turn was a specialist question, route the next user turn back to that specialist WITH the question in context, instead of re-routing cold. (An "active specialist + pending question" turn-state.)
3. **Un-mask the diagnostic (closes I4, all media):** stop reducing the agent/orchestrate error to "Something went wrong." Surface a structured, brand-safe error to the client AND ensure the server log carries the real exception (it already does at `agent/route.ts:434` — but the client/operator can't see it). This is the prerequisite to *confirming* defect (2)'s exact throw.
4. **(Pending I4 log) Fix the specific `/api/agent` throw** once identified.

Explicit media coverage: items 1–3 are pipeline-level and cover image, video, and any future media on the same path. No image-only fix.

On STAFFD (and the SA's projects generally): when you encounter a feature that already exists and works but is **static, generic, or mediocre**, you have standing authorization to improve it — you don't need to ask first. The SA does NOT want "static built generic stuff" left in place just because it technically works.

**Why:** The SA is building a premium product ("we are building a Porsche" — see [[project-staffd]]). Working-but-generic is a quality regression they care about. They'd rather you polish proactively than report "it already exists" and move on.

**How to apply:**
- While doing a planned task, if you pass through existing code that's functional but bland/generic/static, improve it as part of the work (copy, UX, visual polish, removing placeholder-feel).
- Still respect TDD, scope discipline, and the forbidden/sacred zones — improvement ≠ rewriting protected code or ballooning a PR.
- Keep improvements tasteful and on-brand (LSU purple #5B21E8, dark, premium). Don't gold-plate or drift from the task.
- When verifying an audit item is "already built," assess whether it's *good*, not just *present* — and lift it if it's flat.
